The prompt::
You are a senior MNC backend engineer. Build the backend for **Hadi Perfumes** phase-by-phase as a real production system.

Non-negotiable rules:
- No hallucinations.
- No fake code/details.
- No hardcoded business constants.
- Every phase must include unit + integration + e2e tests.
- Keep FTC compliance central: commission only from verified retail sales.

Project identity:
- Name: Hadi Perfumes
- Domain: networked direct-selling MLM + eCommerce
- **Architecture pivot (already implemented in Phase 4): Admin-owned single-vendor catalog**

Critical business reality:
- This is **not** a P2P marketplace.
- Participants do not create stores or list products.
- There are no seller profile, vendor KYC, or marketplace payout-split requirements.
- `listings.seller_id` may exist for FK compatibility, but operationally maps to company/admin owner only.

Core systems:
1) Identity + sponsorship graph
2) Network graph + qualification
3) Admin catalog + atomic inventory
4) Orders + checkout + payment processing
5) Commission + ledger + payout settlement
6) Compliance + fraud + audit

Safest backend rule:
Do NOT pay commissions on:
- signup alone
- recruitment alone
- rank upgrades alone
- self-purchase loops
- fake/duplicate orders

Recommended implementation blueprint:

Phase 1 — Commission & compliance rules
- Versioned immutable policy entities
- Commission/rank qualification constraints
- Admin policy activation + audit logging
- Tests for deterministic rule evaluation

Phase 2 — Identity + referrals
- OTP send/verify/signup
- Referral code validation
- Self-referral and circular-upline blocking
- Onboarding audit trails + anti-abuse checks
- Tests for all referral edge cases

Phase 3 — Network graph + qualification
- Materialized path + adjacency support
- Downline/upline query support
- Rank/qualification recalculation engine
- Admin-safe sponsor correction flow with logs
- Concurrency and invariants tests

Phase 4 — Admin-owned catalog + strict inventory (implemented)
- Admin-only listing/category management
- Listing moderation/status transitions
- Inventory item + reservation model
- Atomic reserve pattern: `UPDATE ... WHERE available_qty >= :qty`
- Reservation TTL + release/expiry jobs
- Concurrency tests to prove no oversell

Phase 5 — Orders, checkout, payments (single merchant)
- Cart + order lifecycle state machine
- Idempotent checkout + payment endpoints
- Standard Stripe charge flow where platform receives 100% of funds
- Webhook signature verification + deduplication
- No Stripe Connect onboarding/transfers/seller splits
- Order/payment reconciliation jobs + tests

Phase 6 — Commission ledger + payouts
- Append-only ledger entries
- Commission events generated from eligible paid retail orders
- Pending→available release after delivery/return windows
- Clawback events for refunds/chargebacks
- Participant commission payouts from platform pool
- Tests proving ledger determinism and reversal safety

Phase 7 — Returns/disputes/fraud
- Return and dispute workflows
- Risk scoring and payout/commission hold hooks
- Admin review tooling + immutable audit logs

Phase 8 — Observability/security/scale
- Metrics/traces/structured logs
- Idempotency keys for all money-moving endpoints
- DLQ/retry strategy for async workers
- RBAC hardening for admin APIs

Data model direction:
Use tables like:
- users, referral_codes, referral_redemptions, sponsorship_links, network_nodes
- compensation_policy_versions, commission_rules, rank_rules
- product_categories, listings, listing_images, inventory_items, inventory_reservations, inventory_events
- orders, order_items, payments, payment_webhook_events
- commission_events, ledger_entries, payout_requests, payout_batches
- refunds, returns, disputes, audit_logs, fraud_signals

Do not introduce:
- seller_profiles
- vendor_kyc
- store_details
- multi-party Stripe Connect splits

Quality bar:
- Migrations first
- DTO validation on all inputs
- Transactional consistency for inventory/orders/ledger
- Full test coverage in every phase before moving forward
